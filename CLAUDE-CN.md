# CLAUDE.md - 中文增强版

> 基于 Andrej Karpathy 的 LLM 编码陷阱观察，针对中文开发者优化

## 核心原则

### 原则1: 先思考，再编码

**不要假设。不要隐藏困惑。暴露权衡。**

LLM 经常默默选择一种解释并继续执行。这个原则强制显式推理：

- **明确陈述假设** — 如果不确定，问而不是猜
- **呈现多种解释** — 当存在歧义时不要默默选择
- **适时反驳** — 如果存在更简单的方法，说出来
- **困惑时停下** — 说明不清楚的地方并寻求澄清

### 原则2: 简单优先

**解决问题的最少代码。不要投机。**

对抗过度工程的倾向：

- 不要添加未被要求的功能
- 不要为单次使用的代码创建抽象
- 不要添加未被要求的"灵活性"或"可配置性"
- 不要为不可能的场景添加错误处理
- 如果 200 行可以变成 50 行，重写它

**测试标准**: 资深工程师会说这太复杂吗？如果是，简化它。

### 原则3: 精确修改

**只改必须改的。只清理自己的混乱。**

编辑现有代码时：

- 不要"改进"相邻的代码、注释或格式
- 不要重构没有问题的东西
- 匹配现有风格，即使你会用不同方式
- 如果注意到无关的死代码，提及它 — 不要删除它

当你的修改产生孤儿时：

- 删除**你的修改**导致未使用的导入/变量/函数
- 不要删除预先存在的死代码，除非被要求

**测试标准**: 每一行修改都应该直接追溯到用户的请求。

### 原则4: 目标驱动执行

**定义成功标准。循环直到验证。**

将命令式任务转换为可验证的目标：

| 不要说... | 转换为... |
|---------|---------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复bug" | "编写重现它的测试，然后让它通过" |
| "重构X" | "确保测试在之前和之后都通过" |

对于多步骤任务，陈述简要计划：

1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]

强成功标准让 LLM 独立循环。弱标准（"让它工作"）需要持续澄清。

## 🇨🇳 中文开发者增强

### 原则5: 中文友好

**尊重中文注释和命名。理解中文语义。**

中文代码处理规则：

- **保留所有中文注释**，除非明确要求修改
- **理解中文变量名和函数名**的语义（如 `用户信息`、`获取数据`）
- **中文字符串不要随意修改**，特别是用户提示和错误信息
- **中文文档格式**遵循国内习惯（如使用全角标点）

示例：

```javascript
// ❌ 错误：删除或修改中文注释
function getData() { ... }

// ✅ 正确：保留中文注释
// 获取用户数据，包含权限验证
function getData() { ... }
```

### 原则6: 国内生态适配

**优先国内方案。考虑网络环境。遵循主流实践。**

国内开发环境优化：

#### 依赖管理
- 优先使用 **淘宝镜像** 或 **npmmirror**
- 推荐 **pnpm** 或 **yarn** 而非 npm
- 注明依赖安装命令（考虑镜像源）

#### 网络优化
- CDN 优先使用国内服务（七牛、阿里云、腾讯云）
- API 调用考虑超时和重试
- 图片/视频资源使用国内 OSS

#### 框架适配
- **Vue 生态**: 遵循 Vue 3 + TypeScript 最佳实践
- **微信小程序**: 遵循微信官方规范
- **Taro/uni-app**: 多端兼容性优先
- **Ant Design/Element Plus**: 遵循组件库规范

#### 代码规范
- 使用 **ESLint + Prettier** 中文配置
- 遵循 **Airbnb** 或 **Standard** 规范
- Git commit 信息支持中文

## 实战案例

### 案例1: Vue 3 + TypeScript 项目

**场景**: 添加用户列表页面

**❌ 错误做法**:
```typescript
// 过度抽象，添加未要求的功能
interface BaseListConfig<T> {
  data: T[];
  pagination: PaginationConfig;
  filters: FilterConfig;
  // ... 10+ 个配置项
}
```

**✅ 正确做法**:
```typescript
// 最小化实现，只满足当前需求
interface UserListProps {
  users: User[];
  onPageChange: (page: number) => void;
}
```

### 案例2: 微信小程序开发

**场景**: 修复登录按钮样式

**❌ 错误做法**:
```css
/* 顺便"优化"了其他样式 */
.login-btn { ... }
.header { ... }  /* 未被要求修改 */
.footer { ... }  /* 未被要求修改 */
```

**✅ 正确做法**:
```css
/* 只修改登录按钮 */
.login-btn {
  /* 修改内容 */
}
```

### 案例3: Node.js 后端服务

**场景**: 添加用户注册接口

**目标驱动方式**:

1. 编写测试用例 → 验证: 测试失败（接口不存在）
2. 实现接口逻辑 → 验证: 测试通过
3. 添加参数验证 → 验证: 无效参数测试通过
4. 添加错误处理 → 验证: 异常场景测试通过

## 效果验证

这些指南有效的标志：

- ✅ **diff 中更少的不必要修改** — 只出现请求的修改
- ✅ **更少因过度复杂而重写** — 代码第一次就简单
- ✅ **实现前先提出澄清问题** — 而不是犯错后
- ✅ **干净、最小的 PR** — 没有顺便重构或"改进"
- ✅ **中文注释和命名被保留** — 除非明确要求修改

## 注意事项

这些指南偏向谨慎而非速度。对于简单任务（简单的拼写错误修复、明显的单行修改），使用判断 — 不是每个修改都需要完整的严格性。

目标是减少非简单工作中的代价高昂的错误，而不是减慢简单任务。
